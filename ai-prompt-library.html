01

Generate a First Draft

Scaffold docs for a new feature or product from scratch

PROMPT
Act as an experienced technical writer with expertise in [INDUSTRY/DOMAIN].

I need to create documentation for [PRODUCT/FEATURE NAME].

Here is the context:

- Target audience: [e.g., developers, end users, IT admins]
- Documentation type: [e.g., user guide, API reference, release notes]
- Key functionality to cover: [LIST 3: 5 BULLET POINTS]
- Tone: [e.g., formal, conversational, instructional]

Write a complete first draft including: an overview section, step-by-step instructions, and a troubleshooting section. Use clear headings, numbered steps for procedures, and plain language appropriate for the audience.
Pro Tip: Provide specific context to improve draft accuracy. Attach existing docs as examples if the platform supports file uploads.

Simplify Complex Technical Content

Rewrite jargon-heavy content for a general audience

PROMPT
You are a technical writer specializing in plain language. Below is a piece of technical content written for expert-level readers. Rewrite it for [TARGET AUDIENCE, e.g., non-technical business users / beginners / customers].

Original content:

[PASTE YOUR CONTENT HERE]

Guidelines:
- Replace all jargon with plain-language equivalents
- Keep sentences under 20 words where possible
- Use active voice throughout
- Preserve all factual accuracy: do not omit critical information
- Maintain the same structure (headings, lists, steps) as the original

Output the rewritten version followed by a short glossary defining any technical terms that could not be fully avoided.
Pro Tip: Describe the audience's exact technical level (for example, 'someone with no IT background') to improve results.

<document>

Peer Review Feedback Generator

PROMPT

Document type: [e.g., API guide, user manual, release notes, README]
Intended audience: [DESCRIBE]
Publishing context: [e.g., public developer portal, internal wiki, help center]

Document:
[PASTE DOCUMENT HERE]

Provide feedback under these headings:
1. Clarity: are instructions clear and unambiguous?
2. Completeness: is anything missing that the reader will need?
3. Accuracy: flag any statements that appear incorrect or need verification
4. Structure: is the information organized in the right order?

09

Readability & Accessibility Audit

Check plain language, accessibility, and global English

PROMPT
You are a technical editor specializing in plain language and accessibility. Audit the document below against these criteria:

Document:
<document>
[PASTE DOCUMENT HERE]
</document>

Audit criteria:
- Readability: flag sentences over 25 words; identify passive voice; highlight unnecessary jargon
- Global English: flag idioms, metaphors, and culturally specific references that will not translate well
- Accessibility: identify spatial language ("see above", "click the icon on the left"), color-only instructions, and non-descriptive links ("click here")
- Inclusivity: flag gendered language; suggest gender-neutral alternatives

Output format:
- One table per criterion
- Columns: Original text | Issue | Suggested fix
- End with a summary paragraph of the 3 highest-priority improvements
Pro Tip: Run this prompt before every external publish to catch easily overlooked accessibility issues.

Document 1: [TITLE/TYPE]:
<document_1>
[PASTE DOCUMENT 1 HERE]
</document_1>

Document 2: [TITLE/TYPE]:
<document_2>
[PASTE DOCUMENT 2 HERE]
</document_2>

1. Terminology: the same concept referred to by different names
2. UI labels: product names, button names, or field names that differ
3. Formatting: date formats, code style, heading capitalization

5. Instructions: conflicting steps or contradictory information

Pro Tip: Paste existing documentation to understand the terminology landscape quickly when onboarding to a new team.

Convert a meeting or interview recording into structured docs

<transcript>

Generate Video Captions and Alt Text

Accessibility-ready captions and image descriptions at scale

PROMPT
You are a technical writer specializing in accessibility. I need two things: cleaned-up captions for a video, and alt text for the screenshots used in associated documentation.

Raw auto-generated captions (unedited):
[PASTE RAW CAPTIONS HERE]

Screenshots to describe (list them in order):
[DESCRIBE EACH SCREENSHOT BRIEFLY, e.g., "Screenshot 1: the login screen with an email field and a blue Sign In button"]

For the captions:
- Correct spelling, punctuation, and speaker identification errors
- Keep timing markers if present
- Do not change the meaning or add information not in the original

For the alt text:
- Write functional alt text that describes what the image shows and why it matters in context
- Keep each description under 125 characters
- Do not start with "Image of" or "Screenshot of"
Pro Tip: Describe the function, not the appearance. Use "Submit button" rather than "blue rounded button."

Convert Markdown to Structured HTML

Output publication-ready HTML with semantic markup

PROMPT
You are a technical writer and web developer. Convert the Markdown document below into semantic, accessible HTML.

Markdown source:
[PASTE MARKDOWN HERE]

Requirements:

- Wrap code samples in pre + code tags with a data-language attribute
- Convert note/tip/warning callouts into aside elements with a class attribute (class="note", class="tip", class="warning")

- Do not add inline styles or classes beyond those specified above
- Output only the HTML body content: no html, head, or body tags
Pro Tip: Use semantic HTML to improve SEO and screen reader compatibility for web-published developer documentation.
17

Localization Prep: Simplify for Translation

Prepare English source text to reduce translation costs

PROMPT
You are a technical writer preparing documentation for translation. Rewrite the document below to minimize localization costs and improve translation accuracy.

Document:
[PASTE DOCUMENT HERE]

Apply these rules:
1. Break all sentences over 20 words into two shorter sentences
2. Remove all idioms, metaphors, and culturally specific expressions. Replace with literal equivalents
3. Replace pronouns where the referent is ambiguous (e.g., replace "it" with the noun it refers to)
4. Use consistent terminology: the same concept must always use the same word
5. Remove contractions (replace "don't" with "do not", "can't" with "cannot")
6. Use the active voice throughout
7. Flag any terms that should NOT be translated (product names, code strings, UI labels) by wrapping them in [NOTRANSLATE]...[/NOTRANSLATE]

Output the revised document followed by a list of flagged no-translate terms.
Pro Tip: Write shorter, clearer source text to reduce translation costs and improve translated quality.

20

Improve Sentence-Level Clarity

Line-edit a passage for tightness, voice, and precision

PROMPT
You are a technical editor performing a line edit. Rewrite each sentence in the passage below to improve clarity, directness, and readability. Do not change the meaning or remove factual content.

Passage:
[PASTE PASSAGE HERE]

Apply these rules sentence by sentence:
- Remove unnecessary words (e.g., "in order to" → "to"; "due to the fact that" → "because")
- Convert passive voice to active voice
- Break sentences over 25 words into two
- Replace vague nouns with specific ones (e.g., "the system" → name the system)
- Remove hedging language where it is not needed (e.g., "it should be noted that")

Output format:
| Original | Revised | Change made |
(One row per sentence changed. Skip unchanged sentences.)
Pro Tip: Apply this prompt to personal writing. Compare the original and revised columns to train your editing skills.
